# LCNC A2A Builder, Specification Document

> Generated on: 2026-04-30
> Project: lcnc-a2a
> Version: 1.0
> Status: Draft
> Type: Greenfield Specification

## 1. Executive Summary

LCNC A2A Builder is a low code / no code web application for designing, hosting, and observing AI agents that expose themselves over the A2A (Agent to Agent) protocol. Each agent designed in the builder becomes a live A2A endpoint at `/agents/<agent_id>` that any A2A compliant client (such as the sibling project `web-a2a`) can invoke. The builder lets a user pick an execution mode (Simple, ReAct, Plan & Execute), write the system prompt(s), attach MCP tool servers (stdio or streamable HTTP), set guardrails, start or stop the agent, and observe live metrics (tokens, requests, average response time, average loop count, time consumed) plus per run execution traces. Tool implementations are out of scope: the project consumes MCP servers but never implements them.

The application reuses the visual identity, theming abstraction, and operational conventions of the sibling `web-a2a` project (IBM Carbon Design System CSS, Python 3.13, FastAPI, HTMX + Jinja2, async SQLAlchemy on PostgreSQL, Fernet encrypted secrets, OpenTelemetry JSONL traces).

## 3. Scope

### 3.1 In Scope

- Multi user web builder, each user owns and manages their own agents.
- Three execution modes: Simple, ReAct (Thought, Act, Observe with embedding similarity stop), Plan & Execute (planner pass + parallel stage executor + replan).
- Agent lifecycle: create, edit, delete, start, stop.
- MCP tool attachment per agent, two transports: stdio (subprocess spawned by the app) and streamable HTTP (URL + optional encrypted headers); tool list discovered automatically via the MCP protocol.
- A2A server endpoint per agent at `/agents/<id>` with SSE streaming, plus the standard `/agents/<id>/.well-known/agent-card.json`.
- Per agent API key authentication for external A2A clients.
- Per agent metrics aggregation (tokens, request count, average response time, average loops) and per run execution traces (Thought, Act, Observe entries; tool inputs/outputs; token usage; similarity scores).
- LLM provider abstraction supporting OpenRouter and any local OpenAI compatible HTTP endpoint, configurable per agent.
- Embedding calls go to the same provider as the agent's LLM (uses `/v1/embeddings`).
- Per A2A `contextId` conversation memory persisted in PostgreSQL.
- Dev mode email login (single field) for the builder UI; pluggable `AuthProvider` ABC for future Google OAuth2.
- Fernet encryption for stored secrets (per agent API keys hash, MCP HTTP headers, per agent provider API keys).
- OpenTelemetry SDK with JSONL file exporter; LLM spans record model, tokens, cost, duration only, never prompts or responses.
- IBM Carbon Design System styling reused from `web-a2a` (colors, fonts, theme tokens).
- Local execution only (pure web app, no Docker for MVP).

### 3.2 Out of Scope (Non Goals)

- Implementation of MCP tool servers themselves.
- Agent versioning, rollback, diff history.
- Import / export of agent JSON definitions.
- Starter templates / cloning library.
- Built in test playground inside the builder (testing happens via external A2A clients such as `web-a2a`).
- Google OAuth2 (deferred; the `AuthProvider` ABC is in scope but only the dev mode implementation is shipped).
- Docker / Cloud Run deployment artifacts (deferred to a later spec).
- Team / organization multi tenancy (each user is isolated; no sharing).
- Cost computation for non OpenRouter providers (token counts only for local endpoints).
- Performance SLOs on agent runs (latency dominated by LLM and MCP).
- Hard performance targets beyond a single dashboard rendering baseline.

## 4. User Personas & Actors

- **Builder User**: A human who signs into the web UI, creates and edits agents, starts and stops them, observes metrics and traces. Owns a private set of agents; cannot see other users' agents.
- **External A2A Client**: Any A2A compliant client (such as `web-a2a`, Claude Desktop with an A2A bridge, or a custom script) that calls `/agents/<id>` with a valid per agent API key. Does not authenticate as a user; authenticates as a caller of one specific agent.
- **MCP Server (external)**: An out of process program that exposes tools via the MCP protocol over stdio or streamable HTTP. Not part of this project; the app consumes it.
- **LLM Provider (external)**: Either OpenRouter (`https://openrouter.ai/api/v1`) or any local server implementing the OpenAI compatible API (chat completions + embeddings). Configured per agent.

## 5. Usage Scenarios

### SC-001: Builder user signs in
**Actor:** Builder User.
**Preconditions:** App is running; PostgreSQL is reachable; user is not currently logged in.
**Flow:**
1. User opens the root URL.
2. App responds with the dev mode login page (single email field + display name field).
3. User submits `alice@example.com`, name `Alice`.
4. App creates or updates the `users` row, issues a session cookie, redirects to the agents dashboard.
**Postconditions:** A `users` row exists for `alice@example.com`; session cookie set; user lands on `/agents`.
**Exceptions:**
- EXC-001a: Empty email submitted ŌåÆ render login page with field error `email_required`, HTTP 400.
- EXC-001b: Email exceeds 255 characters ŌåÆ render error `email_too_long`, HTTP 400.

### SC-002: List own agents with metrics
**Actor:** Builder User.
**Preconditions:** User signed in.
**Flow:**
1. User navigates to `/agents`.
2. App queries the user's agents and aggregated metrics from `agent_runs` (token totals, request count, average duration, average loop count) for the last 30 days.
3. App renders a table: name, mode, model, status (started / stopped), API key fingerprint, requests, tokens in / out, total time, average response time, average loops, last run timestamp.
**Postconditions:** Page rendered with the user's agents only.
**Exceptions:**
- EXC-002a: User has zero agents ŌåÆ render empty state with a `Create agent` call to action.
- EXC-002b: Aggregation query fails ŌåÆ render the agent rows with placeholder `metrics_unavailable` markers, log error.

### SC-003: Create a new agent
**Actor:** Builder User.
**Preconditions:** User signed in.
**Flow:**
1. User clicks `Create agent` on the dashboard.
2. App renders the agent edit form with fields: `name`, `description`, `mode` (Simple / ReAct / Plan & Execute), `model_provider` (openrouter / openai_compatible), `model_endpoint` (URL, defaulted by provider), `model_id`, `provider_api_key`, `system_prompt` (Simple / ReAct only), `planner_prompt` (PE only), `executor_prompt` (PE only), guardrails (`max_loops`, `max_tokens`, `similarity_threshold` for ReAct, `max_steps` for PE), and an MCP tools sub form (initially empty).
3. User fills `name=fin-analyst`, picks `ReAct`, picks `openrouter`, picks model `anthropic/claude-sonnet-4-5`, supplies an OpenRouter API key, pastes the example ReAct system prompt, leaves guardrails at defaults (`max_loops=10`, `max_tokens=8000`, `similarity_threshold=0.95`).
4. User clicks `Save`. App validates, encrypts the provider API key, generates a single `default` API key for external A2A clients, stores the agent with `status=stopped`.
5. App redirects to the agent detail page showing the freshly created agent + the generated API key in plain text once (with a `Copy` action and a warning that it will not be shown again in plain).
**Postconditions:** `agents` row created with `status=stopped`; `agent_api_keys` row created; provider API key stored Fernet encrypted.
**Exceptions:**
- EXC-003a: Missing `name` or `model_id` ŌåÆ form re-rendered with field errors, HTTP 400.
- EXC-003b: Duplicate name for the same user ŌåÆ field error `name_taken`, HTTP 400.
- EXC-003c: PE mode but `planner_prompt` or `executor_prompt` empty ŌåÆ field error `prompts_required`, HTTP 400.
- EXC-003d: `max_steps` outside `[1, 50]` for PE ŌåÆ field error `max_steps_out_of_range`, HTTP 400.

### SC-004: Edit an existing agent
**Actor:** Builder User.
**Preconditions:** User owns at least one agent.
**Flow:**
1. User clicks the agent name in the dashboard.
2. App renders the same edit form prefilled (provider API key shown as `********`, current value preserved unless user types a new one).
3. User changes the system prompt, clicks `Save`.
4. App validates, persists the change, redirects back to the agent detail page with a flash message.
**Postconditions:** `agents.updated_at` advances; new field values persisted; provider API key unchanged when the masked field is left untouched.
**Exceptions:**
- EXC-004a: User attempts to load `/agents/<id_of_other_user>` ŌåÆ HTTP 404 (do not leak existence).
- EXC-004b: Validation errors as in SC-003.
**Cross scenario notes:** When the agent is `started` and an in-flight A2A run is using the prior config, the in-flight run keeps the prior config; new runs after save use the new config.

### SC-005: Stop or start an agent
**Actor:** Builder User.
**Preconditions:** User owns the agent.
**Flow:**
1. From the dashboard or detail page, user clicks `Start` (or `Stop`).
2. App toggles `agents.status`.
3. The route at `/agents/<id>` and `/agents/<id>/.well-known/agent-card.json` becomes active (start) or returns HTTP 503 (stop).
**Postconditions:** `agents.status` updated; in-flight runs are not interrupted; new requests after the toggle observe the new state.
**Exceptions:**
- EXC-005a: Toggle fails because of DB write error ŌåÆ flash error, status unchanged.

### SC-006: Delete an agent
**Actor:** Builder User.
**Preconditions:** User owns the agent.
**Flow:**
1. User clicks `Delete` on the agent detail page.
2. App shows a confirmation modal naming the agent.
3. User confirms.
4. App cancels any in-flight A2A run for that agent, deletes `agents`, cascades to `agent_api_keys`, `agent_mcp_servers`, `agent_runs`, `agent_run_steps`, `agent_contexts`, `agent_messages`.
5. Subsequent calls to `/agents/<id>` return HTTP 404.
**Postconditions:** All rows for the agent removed; cancellation message logged in the trace of any in-flight run.
**Exceptions:**
- EXC-006a: User attempts to DELETE another user's agent ŌåÆ HTTP 404.

### SC-007: Configure MCP tools on an agent
**Actor:** Builder User.
**Preconditions:** User owns the agent.
**Flow:**
1. On the agent edit page, the `Tools` sub form lists currently attached MCP servers.
2. User clicks `Add MCP server`, picks transport `stdio`, provides command `uvx mcp-server-fetch`, optional env vars.
3. User clicks `Discover tools`. App spawns a transient stdio MCP client, completes the MCP handshake, lists the server's tools, displays them in the UI.
4. User clicks `Save`. App stores the server config; the discovered tool list (name, description, JSON schema) is cached on the row.
5. For HTTP transport, user provides `url` and optional headers; same discovery flow.
**Postconditions:** `agent_mcp_servers` row created; `tools_cache` populated; secrets (env values, headers) Fernet encrypted.
**Exceptions:**
- EXC-007a: stdio command exits non zero or hangs > 10 s ŌåÆ discovery fails with `mcp_discovery_failed` and a captured stderr excerpt; nothing persisted.
- EXC-007b: HTTP MCP returns non 2xx or invalid MCP envelope ŌåÆ same error.
- EXC-007c: Transport changed (stdio Ōåö HTTP) on existing server ŌåÆ treated as new config, requires re-discovery before save.

### SC-008: View per agent execution history and traces
**Actor:** Builder User.
**Preconditions:** User owns the agent and at least one A2A request has run.
**Flow:**
1. From the agent detail page user clicks `Runs`.
2. App lists the most recent 100 runs: timestamp, A2A `task_id`, A2A `context_id`, status, duration, loops, tokens in / out, USD cost (if available), summary first 80 chars of final answer.
3. User clicks a run to expand the per loop trace: each step displays role (thought / action / observation / plan / step_result), tool name + arguments + result, token usage, similarity score (ReAct), step status (PE).
**Postconditions:** Page rendered.
**Exceptions:**
- EXC-008a: No runs yet ŌåÆ empty state with hint `Send a message to /agents/<id> to see runs here.`
- EXC-008b: Trace step contains > 4 KB of tool I/O ŌåÆ render truncated with a `View full` action that streams from the DB.

### SC-009: External client invokes a Simple mode agent
**Actor:** External A2A Client.
**Preconditions:** Agent exists, `status=started`, valid API key issued.
**Flow:**
1. Client POSTs an A2A `SendStreamingMessage` to `/agents/<id>` with `Authorization: Bearer <api_key>` and a TextPart containing the user message.
2. App authenticates the API key, loads the agent config, opens the conversation context (creates one if `contextId` not provided), records `agent_runs` row with `status=running`.
3. App calls the LLM once with `system_prompt`, the prior persisted messages for this `contextId`, the user message, and the MCP tool catalog (OpenAI tools format) as the tool list.
4. If the LLM emits a tool call, the executor invokes the MCP tool, appends the tool_result, calls the LLM again, and repeats until the LLM emits a final assistant message with no tool call. (Simple mode loops only on tool calls; there is no separate Thought / Act loop.)
5. App streams SSE `TaskStatusUpdate` (state `working` then `completed`) and `TaskArtifactUpdate` (text deltas) back to the client.
6. App persists messages, the run, the per step trace, increments aggregated metrics.
**Postconditions:** `agent_runs.status=completed`; `agent_messages` rows appended; metrics counters updated.
**Exceptions:**
- EXC-009a: Tool call fails 3 retries ŌåÆ returns `tool_result` with error to the LLM; LLM continues. (See FR-018.)
- EXC-009b: LLM returns 429 or 5xx ŌåÆ propagated to the client as A2A task `failed` with reason `llm_provider_error`.
- EXC-009c: Conversation context exceeds the configured context cap ŌåÆ oldest non system messages dropped; warning emitted in trace.

### SC-010: External client invokes a ReAct mode agent
**Actor:** External A2A Client.
**Preconditions:** Same as SC-009 plus mode is ReAct.
**Flow:**
1. Same intake as SC-009.
2. App enters the ReAct loop: each iteration sends the system prompt + scratchpad (prior thoughts/actions/observations) + user message + tool catalog. The LLM emits either a thought + tool call (continue) or a final answer (stop).
3. After each iteration the executor computes the embedding of the latest LLM textual output and compares it (cosine similarity) to the previous iteration's output. If `similarity >= similarity_threshold` (default 0.95), the loop stops and the previous final candidate is returned with a trace marker `stopped_by_similarity`.
4. Loop also stops on: final answer, `max_loops` reached, total token usage `>= max_tokens`. On guardrail hit, the executor performs one final LLM call asking it to synthesize an answer from the accumulated scratchpad (FR-017).
5. Streaming: SSE per iteration emits a `TaskStatusUpdate` with state `working` and a small JSON envelope `{loop: N, phase: thought|action|observation}`; final `TaskArtifactUpdate` carries the answer.
**Postconditions:** Same as SC-009 plus `agent_runs.loops`, `agent_runs.stop_reason Ōłł {final, similarity, max_loops, max_tokens}` set.
**Exceptions:**
- EXC-010a: Embedding call fails 3 retries ŌåÆ A2A task ends `failed`, reason `embedding_unavailable` (FR-019).
- EXC-010b: Tool call fails 3 retries ŌåÆ tool_result with error returned to the LLM (FR-018).
- EXC-010c: Same loop produces no parseable thought/action ŌåÆ executor records `parse_error` in the trace and treats it as one consumed loop iteration toward `max_loops`.

### SC-011: External client invokes a Plan & Execute agent
**Actor:** External A2A Client.
**Preconditions:** Same as SC-009 plus mode is Plan & Execute.
**Flow:**
1. Same intake as SC-009.
2. **Planner pass**: app sends `planner_prompt` (rendered with the actual MCP tool catalog) + the user message; expects strict JSON `{goal, steps[]}`. On parse / schema failure, executor retries the planner once with the validation error, then fails with `planning_failed` (FR-020).
3. App validates: `1 <= len(steps) <= max_steps` (configurable per agent, default 20). On violation, retry once then fail.
4. **Executor pass**: app groups steps by `stage`. For each stage, it runs all member steps in parallel via `asyncio.gather`, each step using `executor_prompt` rendered with the step description, tool, args, success criterion, and resolved `${step_N.output}` references.
5. Executor returns one of `success`, `failure`, `replan_requested`. On `failure` the run aborts with `step_failed`. On `replan_requested` the planner is invoked again with the partial outputs and the requested reason; the remaining steps are replaced. Up to 3 replans per A2A task; on the 4th, the run fails with `replan_exceeded`.
6. After all steps complete, app calls the LLM one final time with the prompt "Synthesize the answer from these step outputs" and streams the answer.
7. SSE: a `TaskStatusUpdate` with `{phase: planning}` is emitted on planner start, then per stage `{phase: executing, stage: N, steps: [ids]}`, then `{phase: synthesizing}`, then `TaskArtifactUpdate` carries the answer.
**Postconditions:** Same as SC-009 plus the plan JSON is persisted in `agent_runs.plan` and per step results in `agent_run_steps`.
**Exceptions:**
- EXC-011a: Planner returns invalid JSON twice ŌåÆ `planning_failed`.
- EXC-011b: Planner returns a step referencing an unknown tool ŌåÆ validation rejects the plan, retry once with the tool catalog re-injected.
- EXC-011c: Step references `${step_N.output}` where step N has not run yet (forward dep) ŌåÆ validation rejects the plan, retry once.
- EXC-011d: Tool call within a step fails 3 retries ŌåÆ step status `failure`, returned to executor, executor decides per its rules (FR-018).
- EXC-011e: `max_loops` here applies to the synthesis phase only (never expected to loop, defensive cap = 1); `max_tokens` is a hard ceiling on total LLM tokens consumed across planner + executor + synthesis (default 16000). On hit, force a synthesis call from accumulated step outputs (FR-017).

### SC-012: External client fetches the Agent Card
**Actor:** External A2A Client.
**Preconditions:** Agent exists.
**Flow:**
1. Client GETs `/agents/<id>/.well-known/agent-card.json`.
2. If `agents.status=started`, app returns 200 with a JSON Agent Card containing: `name`, `description`, `version`, `url` (`/agents/<id>`), `capabilities` (`{streaming: true, pushNotifications: false}`), `securitySchemes` (`{bearer_api_key: {type: http, scheme: bearer}}`), `skills` (one entry, name = agent name, description = agent description, tags = mode), `defaultInputModes`, `defaultOutputModes`.
3. If `agents.status=stopped`, app returns HTTP 503 with body `{error: "agent_stopped"}`.
4. If agent does not exist, app returns HTTP 404.
**Postconditions:** None.
**Exceptions:** None beyond above.

## 6. Functional Requirements

### FR-001: Dev mode email login
- **Description:** App provides a `/login` page accepting an email and a display name. On submit, the app upserts a `users` row keyed by email (lowercased), issues a signed session cookie (itsdangerous URLSafeSerializer), and redirects to `/agents`.
- **Inputs:** `email` (string, 1..255 chars, valid RFC 5321 format), `name` (string, 1..255 chars).
- **Outputs:** HTTP 302 to `/agents` with a `Set-Cookie: session=<signed>` header.
- **Business Rules:** Email comparison is case insensitive on the lowercased form. Session expires after 24 hours. The `AuthProvider` ABC exists in code (with a `dev_mode` implementation) so future Google OAuth2 plug in does not require route changes.
- **Priority:** Must.

### FR-002: List user's agents with aggregated metrics
- **Description:** GET `/agents` returns the dashboard listing the current user's agents plus aggregated metrics over the last 30 days (configurable via `LCNC_A2A_METRICS_WINDOW_DAYS`).
- **Inputs:** Session cookie identifying the user.
- **Outputs:** HTML page with the agent table.
- **Business Rules:** Aggregations: `requests = count(agent_runs)`, `tokens_in = sum(agent_runs.tokens_in)`, `tokens_out = sum(agent_runs.tokens_out)`, `avg_duration_ms = avg(agent_runs.duration_ms)`, `avg_loops = avg(agent_runs.loops)`, `total_time_ms = sum(agent_runs.duration_ms)`, `last_run_at = max(agent_runs.started_at)`. Cross user data must never appear.
- **Priority:** Must.

### FR-003: Create a new agent
- **Description:** POST `/agents` creates an agent owned by the current user. GET `/agents/new` renders the form.
- **Inputs:** `name` (1..120 chars, unique per user), `description` (0..2000 chars), `mode Ōłł {simple, react, plan_execute}`, `model_provider Ōłł {openrouter, openai_compatible}`, `model_endpoint` (URL), `model_id` (1..200 chars), `provider_api_key` (string, encrypted at rest), `system_prompt` (Simple / ReAct only, 1..40000 chars), `planner_prompt` (PE only, 1..40000 chars), `executor_prompt` (PE only, 1..40000 chars), `max_loops` (1..50, default 10 for ReAct, irrelevant for Simple, defensive 1 for PE synthesis), `max_tokens` (100..200000, defaults: Simple 8000, ReAct 8000, PE 16000), `similarity_threshold` (0.50..0.99, default 0.95, ReAct only), `max_steps` (1..50, default 20, PE only).
- **Outputs:** HTTP 302 to `/agents/<id>` plus a generated API key shown once.
- **Business Rules:** A single API key labelled `default` is generated on creation, stored as `sha256` hash + last 4 plain in `agent_api_keys`. The plain key is shown exactly once on the response page. Provider API key is Fernet encrypted in `agents.provider_api_key_enc`. Initial `status=stopped`.
- **Priority:** Must.

### FR-004: Edit an agent
- **Description:** GET `/agents/<id>/edit` renders the prefilled form. POST `/agents/<id>` updates the agent.
- **Inputs:** All FR-003 fields. `provider_api_key` left blank means "unchanged".
- **Outputs:** HTTP 302 to `/agents/<id>` with a flash message.
- **Business Rules:** Edits do not affect in-flight runs (each run snapshots config at start). Changing `mode` between PE and others requires the corresponding prompts to be present.
- **Priority:** Must.

### FR-005: Delete an agent
- **Description:** DELETE `/agents/<id>` (or HTMX form POST with `_method=DELETE`) removes the agent.
- **Inputs:** Path id, session cookie.
- **Outputs:** HTTP 302 to `/agents`.
- **Business Rules:** Cascades to `agent_api_keys`, `agent_mcp_servers`, `agent_runs`, `agent_run_steps`, `agent_contexts`, `agent_messages`. In-flight runs are cancelled (cancellation token raised inside the executor).
- **Priority:** Must.

### FR-006: Toggle agent state (start / stop)
- **Description:** POST `/agents/<id>/start` sets `status=started`. POST `/agents/<id>/stop` sets `status=stopped`.
- **Inputs:** Path id, session cookie.
- **Outputs:** HTTP 302 back to the referer (or `/agents`).
- **Business Rules:** State change is immediate and atomic (single UPDATE). In-flight runs are NOT interrupted by stop.
- **Priority:** Must.

### FR-007: Generate per agent API keys
- **Description:** Each agent has at least one API key. Endpoint POST `/agents/<id>/keys` generates a new key (plain shown once, hash stored). Endpoint DELETE `/agents/<id>/keys/<key_id>` revokes a key.
- **Inputs:** Optional `label` (1..60 chars, default `default`).
- **Outputs:** HTML partial showing the new key once.
- **Business Rules:** Keys are 32 random bytes base64url encoded (Ōēł43 chars). Storage: `sha256(key)` plus the last 4 chars in plain (for fingerprint display). At least one key must remain on each agent (revoking the last key is rejected).
- **Priority:** Must.

### FR-008: Configure MCP tools (per agent)
- **Description:** Sub form on the agent edit page. Endpoints: GET `/agents/<id>/mcp` (list), POST `/agents/<id>/mcp` (add), POST `/agents/<id>/mcp/<server_id>/discover` (re run discovery), DELETE `/agents/<id>/mcp/<server_id>`.
- **Inputs (stdio):** `command` (string, full command line incl. args), `env` (map of name ŌåÆ value, values encrypted), `cwd` (optional). **Inputs (http):** `url` (https URL), `headers` (map, values encrypted), `transport` (`streamable_http`, only value supported).
- **Outputs:** HTML partial with the discovered tool list and a save button.
- **Business Rules:** Discovery times out at 10 s. Discovery spawns the stdio process in a sandbox (subprocess with the user's env scrubbed except the explicit `env` map), runs `initialize` then `tools/list`, terminates the process. The cached tool list is `{tools: [{name, description, input_schema}]}`. Encryption uses Fernet with `LCNC_A2A_ENCRYPTION_KEY`.
- **Priority:** Must.

### FR-009: Persist per agent execution traces
- **Description:** Each A2A request creates one `agent_runs` row plus N `agent_run_steps` rows. Steps are appended live during execution.
- **Inputs:** Internal (executor emits step events).
- **Outputs:** Persisted rows.
- **Business Rules:** Step shape: `{role Ōłł {thought, action, observation, plan, step_result, synthesis, error}, content (text), tool_name (nullable), tool_args_json (nullable, max 64 KB), tool_result_json (nullable, max 64 KB), tokens_in (int), tokens_out (int), similarity_to_prev (float, nullable, ReAct only), duration_ms (int), occurred_at (timestamp), seq (int)}`. Tool I/O fields > 4 KB are flagged for truncated rendering in the UI but stored in full up to 64 KB; beyond 64 KB the executor stores a SHA256 hash and a `truncated=true` marker.
- **Priority:** Must.

### FR-010: A2A endpoint per agent
- **Description:** `/agents/<id>` accepts A2A `SendStreamingMessage` (POST, content type `application/json`) and responds with `text/event-stream` (SSE).
- **Inputs:** A2A request envelope including `message` (TextPart, FilePart), optional `contextId`, optional `taskId`.
- **Outputs:** SSE stream of `TaskStatusUpdate` and `TaskArtifactUpdate` events per the A2A protocol.
- **Business Rules:** Authentication via `Authorization: Bearer <key>` resolved against `agent_api_keys`. If `agents.status=stopped`, return HTTP 503. If agent does not exist, return HTTP 404. The same `contextId` reuses prior persisted messages; missing `contextId` triggers creation of a new context.
- **Priority:** Must.

### FR-011: Agent Card endpoint
- **Description:** `/agents/<id>/.well-known/agent-card.json` returns the Agent Card JSON.
- **Inputs:** None.
- **Outputs:** JSON per the A2A Agent Card schema.
- **Business Rules:** When `agents.status=stopped`, return HTTP 503 with `{error: "agent_stopped"}`. When agent does not exist, return HTTP 404. Card content is generated from agent config; `securitySchemes` always declares `bearer_api_key`.
- **Priority:** Must.

### FR-012: API key authentication for A2A clients
- **Description:** Every call to `/agents/<id>` (POST) requires a valid Bearer token.
- **Inputs:** `Authorization: Bearer <key>` header.
- **Outputs:** HTTP 401 with `{error: "auth_required"}` if missing; 403 with `{error: "auth_invalid"}` if the key does not match an active key for THIS agent (keys for OTHER agents are rejected even if otherwise valid).
- **Business Rules:** Lookup is `sha256(key) = stored_hash AND agent_id = path_agent_id`. Constant time comparison.
- **Priority:** Must.

### FR-013: 503 for stopped agents
- **Description:** Both `/agents/<id>` (POST) and `/agents/<id>/.well-known/agent-card.json` return HTTP 503 with body `{error: "agent_stopped"}` when `agents.status=stopped`.
- **Inputs:** N/A.
- **Outputs:** HTTP 503.
- **Business Rules:** Authentication is checked first on POST; an unauthenticated request to a stopped agent still returns 401 (do not leak status to anonymous callers). Authenticated request ŌåÆ 503.
- **Priority:** Must.

### FR-014: Simple mode executor
- **Description:** Executor that loops only on tool calls and emits a final answer when the LLM stops calling tools.
- **Business Rules:** Hard cap on tool call iterations: `max_loops = 50` (defensive, not user configurable for Simple). Tool retry policy per FR-018. Streaming per FR-010.
- **Priority:** Must.

### FR-015: ReAct mode executor
- **Description:** Executor that runs the Thought ŌåÆ Act ŌåÆ Observe loop.
- **Business Rules:** Each iteration computes the cosine similarity between the embedding of the current iteration's textual output and the previous iteration's. If `similarity >= similarity_threshold`, stop with `stop_reason=similarity` and return the previous candidate. Loop also stops on final answer (no tool call), `max_loops` (FR-017), `max_tokens` (FR-017). Embedding model is the provider's default (OpenRouter: `openai/text-embedding-3-small`; openai_compatible: configurable on the agent, default `text-embedding-3-small`). Tool retry per FR-018. Embedding retry per FR-019.
- **Priority:** Must.

### FR-016: Plan & Execute mode executor
- **Description:** Executor that runs a planner pass producing strict JSON, then executes steps in stage parallel batches, with replan support.
- **Business Rules:** Planner is rendered with the agent's MCP tool catalog injected into the `Available tools` section of the planner prompt. Plan JSON validated against schema `{goal: str, steps: [{id: int, stage: int, description: str, tool: str | "synthesize", args: object, success_criterion: str, depends_on: [int]}]}`. Validation: every `tool` must be in catalog or be the literal `synthesize`; every `id` in `depends_on` must reference a step with a strictly lower `stage`; `1 <= len(steps) <= max_steps`. On validation failure, planner is retried ONCE with the error message inlined; second failure ends the run with `planning_failed`. Steps within the same stage execute via `asyncio.gather`. Executor outputs `{step_id, status, output, notes}`. On `replan_requested`, the planner is invoked again with the partial outputs and the reason; up to 3 replans per task; the 4th triggers `replan_exceeded`. After all steps complete, a synthesis call is made.
- **Priority:** Must.

### FR-017: Guardrails (max_loops, max_tokens, force synthesis on hit)
- **Description:** Every executor checks the cumulative loop count and the cumulative LLM token count after each iteration. If either ceiling is hit before a final answer, the executor performs ONE final LLM call asking it to synthesize an answer from the accumulated scratchpad / step outputs.
- **Business Rules:** The synthesis call itself is exempt from `max_loops` but its tokens count toward `max_tokens`. If the synthesis call would exceed `max_tokens` by more than 50 %, it is skipped and the run ends `failed` with reason `guardrail_exceeded_no_synthesis`.
- **Priority:** Must.

### FR-018: MCP tool retry policy
- **Description:** Each tool call attempt that fails (timeout, non zero exit, transport error, malformed MCP envelope) is retried up to 3 times with exponential backoff (200 ms, 600 ms, 1800 ms). After the 3rd retry the failure is surfaced to the LLM as a `tool_result` with `is_error=true` and a structured error message; the agent run continues.
- **Business Rules:** Retries apply to a single tool call invocation, not across iterations. Tool execution timeout per call is 30 s (configurable per MCP server, 1..300 s).
- **Priority:** Must.

### FR-019: Embedding retry policy
- **Description:** Each embedding call (used only by ReAct's similarity stop) is retried up to 3 times with the same backoff schedule as FR-018. After the 3rd retry the agent run ends `failed` with reason `embedding_unavailable`.
- **Business Rules:** Retry only on transport / 5xx / 429. 4xx responses other than 429 fail immediately without retry.
- **Priority:** Must.

### FR-020: Plan validation and bounded retry
- **Description:** When the planner output fails JSON parsing or schema validation, the planner is invoked exactly ONCE more with the validation error inlined. Second failure ends the run with `planning_failed`. `max_steps` is configurable per agent (1..50, default 20).
- **Priority:** Must.

### FR-021: Per context conversation memory
- **Description:** Each A2A `contextId` keeps its message history persisted in `agent_messages`. New messages on the same `contextId` are appended; the executor builds the LLM input from the prior persisted messages plus the new user message.
- **Business Rules:** Soft cap of 50 messages per context for LLM input (oldest non system messages are dropped from the LLM payload but retained in the DB). Hard cap of 1000 messages per context (further appends fail with `context_full`).
- **Priority:** Must.

### FR-022: Token / cost tracking
- **Description:** After each LLM response the executor records `tokens_in`, `tokens_out`, and `cost_usd`.
- **Business Rules:** OpenRouter: read `usage.prompt_tokens`, `usage.completion_tokens`, and `usage.cost` from the response (OpenRouter exposes per request cost). openai_compatible: read `usage.prompt_tokens`, `usage.completion_tokens`; `cost_usd` is NULL. Aggregations on the dashboard treat NULL costs as zero and display `n/a` if any constituent run has a NULL cost in the window.
- **Priority:** Must.

### FR-023: Encrypt secrets at rest
- **Description:** `agents.provider_api_key_enc`, `agent_mcp_servers.env_enc`, `agent_mcp_servers.headers_enc` are Fernet encrypted using `LCNC_A2A_ENCRYPTION_KEY`. `agent_api_keys.key_hash` is SHA 256 (not reversible).
- **Business Rules:** Encryption key is loaded from env at startup; missing key blocks startup with a clear error. API responses NEVER return decrypted secrets (provider key, MCP env values, MCP headers). The masked field convention `********` is used in HTML forms.
- **Priority:** Must.

### FR-024: OpenTelemetry JSONL export
- **Description:** Same OTel pattern as `web-a2a`: a custom JSONL file exporter writes spans to `traces/lcnc-a2a.jsonl` (path configurable via `LCNC_A2A_TRACE_FILE`).
- **Business Rules:** Spans: `db.<operation>`, `mcp.<operation>`, `llm.chat`, `llm.embed`, `executor.<mode>.iter`, `a2a.request`. `llm.*` span attributes record `model`, `provider`, `tokens.prompt`, `tokens.completion`, `cost.usd`, `duration.ms`. Attributes MUST NEVER include the prompt text, the response text, the user message, the tool I/O text, the API key, or any header value. Sensitive fields are explicitly redacted at the exporter boundary.
- **Priority:** Must.

### FR-025: IBM Carbon Design System UI
- **Description:** Reuse the `web-a2a` `carbon.css` file (or a slim subset) and the same theme abstraction.
- **Business Rules:** Copy `web-a2a/src/agent_chat/static/css/carbon.css` and the `themes/` package, rename namespace to `lcnc_a2a`. Same dark mode default (`g100`), light mode (`g10`). Token names from `ThemeTokens` dataclass preserved. Pages MUST visually match `web-a2a` (font: IBM Plex Sans / IBM Plex Mono; interactive blue; same spacing scale).
- **Priority:** Must.

### FR-026: Server side rendering with Jinja2 + HTMX
- **Description:** Same template + partial pattern as `web-a2a`: Jinja2 templates in `templates/`, with `partials/` for HTMX swaps.
- **Business Rules:** All builder POST endpoints return HTML partials suitable for HTMX targets. Full page renders only on initial GET. CSRF tokens via `itsdangerous`, attached to all state mutating forms.
- **Priority:** Must.

## 7. Non Functional Requirements

### 7.1 Performance
No performance SLOs for MVP except a single baseline:
- The agents dashboard (FR-002) MUST render in under 500 ms p95 with 50 agents seeded and 1000 historical runs (PostgreSQL local, dev hardware).
- A2A SSE stream MUST emit its first event within 200 ms after the LLM connection is established (i.e., the app's overhead must not delay streaming).
Other paths (agent runs) are bounded by LLM and MCP latency and have NO target.

### 7.2 Security
- Builder authentication: dev mode email login (FR-001). Pluggable `AuthProvider` ABC for future Google OAuth2; OAuth2 itself is out of scope.
- A2A authentication: per agent Bearer API key (FR-007, FR-012); constant time comparison; sha256 storage.
- Secrets at rest: Fernet encryption (FR-023); key loaded from env, missing key blocks startup.
- CSRF: signed token on all state mutating builder forms (FR-026).
- Cross user isolation: all DB queries scoped by `user_id`; attempts to access another user's agent return 404 (do not leak existence).
- Trace privacy: LLM traces (FR-024) MUST NOT contain prompts, responses, user messages, tool I/O text, or any header / key value. Only metadata (model, token counts, cost, duration).
- No external attack surface beyond the LLM provider URL (configurable per agent), the MCP HTTP URL (configurable per agent), the local stdio MCP processes (subprocess sandboxed with scrubbed env).

### 7.3 Usability
- All builder pages MUST be fully usable on a 1280├Ś720 desktop viewport.
- Carbon Design System look and feel matching `web-a2a` (FR-025).
- Empty states with explicit calls to action.
- Inline form validation errors with the field highlighted.
- Plain text API keys shown exactly once at generation, with a `Copy` action and an explicit warning.
- No accessibility certification target for MVP; only obey Carbon Design defaults.

### 7.4 Reliability
- No uptime target (local tool).
- App MUST recover from crash without data loss (PostgreSQL is durable; in flight A2A runs are abandoned and marked `failed` on next startup via a startup sweep).
- Startup sweep: on boot, any `agent_runs.status='running'` is updated to `status='failed'`, `stop_reason='app_restart'`.

### 7.5 Observability
- **Collector:** JSONL file (default).
- **Trace file path:** `traces/lcnc-a2a.jsonl` (override via `LCNC_A2A_TRACE_FILE`).
- **What to trace:** A2A requests (INFO), DB operations (DEBUG), MCP tool discovery (INFO), MCP tool calls (DEBUG), LLM calls (INFO), LLM embeddings (DEBUG), executor iterations (INFO), errors (ERROR), warnings (WARNING).
- **LLM tracing:** `model`, `provider`, `tokens.prompt`, `tokens.completion`, `cost.usd`, `duration.ms`, `request_id` (for correlation). NEVER prompts, responses, or tool I/O.
- **Custom sub metrics:** per executor iteration: `loop`, `phase`, `similarity_score` (ReAct only). Per PE step: `step_id`, `stage`, `status`. Per MCP tool: `server_id`, `tool_name`, `attempt`, `latency_ms`, `error_class`.
- **Sensitive data exclusion:** prompts, responses, user messages, tool I/O, API keys, OAuth tokens, any header value, the Fernet key. Enforced by a span attribute filter at the exporter boundary.

### 7.6 Deployment
- Local web app, no Docker for MVP.
- Run via `make run-frontend` (FastAPI / uvicorn on port 8080).
- PostgreSQL provisioned by the user (any PG 14+; local socket or TCP).
- Configuration: env vars with prefix `LCNC_A2A_` via `pydantic-settings`.
- Required env vars: `LCNC_A2A_DATABASE_URL`, `LCNC_A2A_ENCRYPTION_KEY`, `LCNC_A2A_SESSION_SECRET`. Optional: `LCNC_A2A_TRACE_FILE`, `LCNC_A2A_METRICS_WINDOW_DAYS`, `LCNC_A2A_DEFAULT_TOOL_TIMEOUT_S`.
- Migrations via Alembic (`make db-migrate`), same pattern as `web-a2a`.

### 7.7 Scalability
No scalability target. Single user concurrent A2A requests must work; concurrent in-flight runs across agents must not block one another (asyncio).

## 16. Glossary

- **A2A**: Agent to Agent protocol; a streaming JSON / SSE protocol for invoking remote AI agents. Defined externally.
- **Agent Card**: JSON metadata document at `/.well-known/agent-card.json` describing an A2A agent's capabilities, skills, and security schemes.
- **MCP**: Model Context Protocol; a protocol for exposing tools to LLMs over stdio or streamable HTTP.
- **Simple mode**: Executor that performs zero or more tool calls in a single LLM driven loop, then returns a final answer. No reasoning loop beyond what the LLM does naturally.
- **ReAct mode**: Executor that explicitly alternates Thought, Act (tool call), Observe (tool result) iterations, with an embedding similarity stop condition between consecutive iterations.
- **Plan & Execute mode**: Executor that first produces a strict JSON plan (planner pass), then runs steps in stage parallel batches with replan support (executor pass), then synthesizes a final answer.
- **Stage**: An integer label on a Plan & Execute step indicating which batch of independent steps it belongs to. Steps with the same stage execute concurrently.
- **Replan**: A signal returned by the executor for a step (`status=replan_requested`) asking the planner to revise the remaining plan.
- **Guardrail**: A bound on loops or total LLM tokens beyond which the executor stops and forces a synthesis call.
- **Trace step**: A row in `agent_run_steps` representing one observable event during a run (a thought, an action, an observation, a plan, a step result, a synthesis call, an error).
- **Per agent API key**: A bearer token issued by the builder for an agent; required by external A2A callers to invoke `/agents/<id>`.
- **Context (A2A)**: A conversation thread keyed by `contextId`; persisted message history reused across invocations of the same `contextId`.
