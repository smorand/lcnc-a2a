# Project Backlog

> Ideas and features deferred for future exploration. Managed by the add-backlog agent.

| ID | Title | Category | Description | Added | Source |
|----|-------|----------|-------------|-------|--------|
| BL-001 | Pluggable memory strategies (summary, vector, per-user partitioning) | Feature | Extend `services/messages.py` beyond the current window-buffer (soft 50 / hard 1000 cap) with three pluggable strategies: summary memory (LLM-generated rolling summary as the cap approaches), vector memory (semantic retrieval via existing `llm/embeddings.py` + pgvector), and per-end-user partitioning inside a single `agent_contexts` row. Requires a new extension point in the memory layer and likely a new DB column or companion table to store summaries and embedding vectors. | 2026-05-10 | manual |
| BL-002 | Enterprise OAuth2 authentication (Google Workspace, Azure AD / Entra ID, IBMid) | Feature | Graft concrete OAuth2 providers onto the existing `AuthProvider` ABC (`auth/`) so that production deployments can replace `DevModeAuthProvider` without touching route code. Providers to support at minimum: GCP (Google Workspace), Azure AD / Entra ID, IBM Cloud / IBMid. Configuration is fully env-var-driven; a single FastAPI callback route handles all providers; the callback maps the email claim to a `User` row (create-on-first-login). The spec already calls out this extension point; this item scopes the full implementation. | 2026-05-10 | manual |
| BL-003 | Observability and cost dashboard per agent and per end-user | Feature | Build a read-only analytics page (Jinja2 + HTMX, Carbon design, consistent with `templates/agents/`) that aggregates the data already stored in `agent_runs` (tokens_in, tokens_out, cost_usd, latency, stop_reason). Metrics: runs per day, token consumption, cost USD, p50/p95 latency, failure rate, broken down by agent and by end-user identity (API key or A2A caller). Add per-agent cost quota configuration and a threshold alert mechanism (email or webhook). | 2026-05-10 | manual |
| BL-004 | Graph execution mode + agent versioning with A/B and canary routing | Feature | Add a fourth execution mode (alongside simple / react / plan_execute): an explicit node graph where the user wires together LLM-call, tool-call, branch, and merge nodes, similar to LangGraph's model. In the same initiative, introduce agent versioning (draft / published states, immutable config snapshots building on the existing `config_snapshot` field in `agent_runs`), version promotion, and A/B or canary traffic splitting on the A2A endpoint (X% to version v1, Y% to v2) with per-run routing decision persisted for comparative analysis. | 2026-05-10 | manual |
| BL-005 | Native RAG and GraphRAG pipeline with pgvector and docling | Feature | Add a document ingestion pipeline (PDF, DOCX, URL) using docling (already available locally as `doc-convert`) and optionally docling-graph for GraphRAG extraction. Store vectors in pgvector on the existing PostgreSQL instance. Expose retrieval to agent runtimes as a first-class MCP-style tool ("retrieve as tool") so all three executors can call it without changes to the dispatch loop. A "knowledge base" is attachable to an agent the same way a MCP server is. Provide upload UI with indexing status inside the agent detail page. | 2026-05-10 | manual |
| BL-006 | HITL approval timeout with auto-policy and notification on INPUT_REQUIRED | Enhancement | Extend the existing `pending_action` / `INPUT_REQUIRED` flow (already used for `destructiveHint` tools) with a configurable timeout per agent and optionally per tool. When the timer expires without a human response, the system applies a configured policy (auto-deny or auto-approve) and fires a notification to the requester (email or outbound webhook). The resolution and the auto-policy decision are persisted in `agent_runs` as an explicit `stop_reason` value so runs history reflects what happened. | 2026-05-10 | manual |
| BL-007 | Agent export as autonomous standalone project (LangGraph, CrewAI, Pydantic AI) | Feature | Add an "Export" button on the agent detail page that generates a downloadable zip containing: self-contained Python agent code in the user's chosen framework (LangGraph at minimum, CrewAI and Pydantic AI as options), `pyproject.toml`, `Dockerfile` and `docker-compose.yml`, a `README.md` with run instructions, and a config file listing env vars to provide (provider API key, MCP server URLs, prompts). The exported agent exposes the same A2A endpoint as its hosted counterpart and has zero runtime dependency on this webapp, allowing users to graduate from the builder to their own infrastructure. | 2026-05-10 | manual |
